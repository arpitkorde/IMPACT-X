# Contributing to IMPACT-X

First — thank you. Every contribution makes this framework more honest, more useful, and more defensible in the real world.

IMPACT-X is built on a simple belief: frameworks without field testing are just philosophy. Your experience, your data, and your challenges to the framework are what turn this from an idea into a standard.

---

## Ways to Contribute

You don't need to be a developer to contribute meaningfully to IMPACT-X.

### 1. Share a Case Study
Run a pilot — even informally, even imperfectly — and share what you found. Did the scores feel right? Where did they break down? What did you learn?

Use the template in [/case-studies/template.md](case-studies/template.md). We welcome null results and failures as much as successes. Honest data is the only kind that helps.

### 2. Build a Vertical Adaptation
The framework is designed for adaptation. If you work in healthcare, education, manufacturing, legal, finance, or any other industry — you have knowledge we don't. Help us build the vertical-specific weight templates, Work Zone examples, and scoring guidance for your field.

Navigate to [/verticals](verticals/) and follow the README in the relevant folder.

### 3. Score an AI Agent
Add an entry to the Agent Registry with your real-world IMPACT scores for an AI tool you actually use on real tasks. See [/agents/registry](agents/registry/) for format guidance.

### 4. Improve the Formulas
If you have data that suggests a scoring formula is wrong, incomplete, or gameable — open an issue and show your work. We will take it seriously.

### 5. Challenge the Framework
If you think a dimension is missing, wrong, or philosophically flawed — say so. Open an issue titled "Framework Challenge:" and make your argument. The best challenges become the next version.

### 6. Translate
IMPACT-X should work in every language and every culture. If you can translate the core documentation, please do.

## Contribution Process

### For documentation, templates, and case studies:
1. Fork the repository
2. Create a branch: `git checkout -b your-contribution-name`
3. Make your changes
4. Submit a pull request with a clear description of what you changed and why

### For formula changes or framework challenges:
1. Open an issue first — describe the problem before proposing the solution
2. Allow 7 days for community discussion
3. If there is broad agreement, submit a pull request referencing the issue

### For agent registry entries:
1. Copy the registry template
2. Fill in your scores with honest methodology notes
3. Submit as a pull request — entries are accepted if methodology is documented, regardless of whether the scores are high or low

## Principles for Contributors

**Be honest before being positive.** A case study that shows the framework didn't work is more valuable than one that polishes the results. We need to know where this breaks.

**Show your methodology.** Scores without methodology are opinions. Scores with methodology are data. Always document how you measured.

**Respect the Fairness Protocol.** Any contribution that suggests measuring people without ensuring access and training first will not be merged. This is non-negotiable.

**Challenge with evidence.** You are welcome to disagree with any dimension, formula, or principle in the framework — but bring data or a reasoned argument. "I don't like this" is not a pull request.

**Credit your context.** If your contribution is specific to an industry, role type, or organisation size — say so. A finding from a 10-person startup and a finding from a 50,000-person corporation are both valuable, but they're not the same.
